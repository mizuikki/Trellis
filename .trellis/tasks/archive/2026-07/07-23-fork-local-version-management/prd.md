# Fork-local version management

## Goal

Make this Trellis fork self-contained: project-template compatibility, the
running CLI version, migrations, and release tooling must be governed by a
fork-owned version line. The fork may selectively import upstream changes but
must neither check the upstream npm package for updates nor publish to npm.

## Confirmed Facts

- This repository is itself a dogfood Trellis project whose deployed template
  stamp is `.trellis/.version` (`0.6.2`). That deployment is not a target of
  this task.
- The CLI and core source packages are both `0.6.7`; this is the source
  version used to deploy Trellis to other projects.
- CLI startup compares `.trellis/.version` with the running CLI's `VERSION`.
- Generated session context invokes `trellis --version` from `PATH`, extracts
  a version or update banner, and compares it with `.trellis/.version`.
- `trellis update` requests the upstream npm `latest` endpoint and reports the
  project, CLI, and npm versions.
- `trellis upgrade`, release preflight, publish CI, manifests, package names,
  docs, and tests contain upstream npm assumptions.
- Migration selection is ordered by the shared semver comparator. A fork
  version scheme therefore cannot be an ambiguous cosmetic suffix.
- The repository has distinct `origin` (the fork) and `upstream` remotes.

## Requirements

- R1: Define a monotonically increasing fork version line, beginning at
  `1.0.0`, as the compatibility version for the CLI, core package,
  `.trellis/.version`, and all future fork migration manifests. Historical
  `0.6.x` manifests remain an ingress chain for the one-time upgrade only.
- R2: Preserve local project-versus-running-CLI mismatch detection and the
  existing downgrade/migration safety guarantees.
- R3: Remove runtime requests and user-facing upgrade guidance that target
  `@mindfoldhq/trellis` or the public npm registry.
- R4: Replace npm-publication release checks and CI behavior with checks that
  remain meaningful for the fork, including the CLI/core version lockstep.
- R5: Record upstream provenance separately from fork compatibility versioning,
  including the upstream baseline and selectively imported commits or changes.
- R6: Make the generated runtime/session behavior identify the fork CLI rather
  than accidentally resolving an upstream `trellis` executable from `PATH`.
- R7: Update relevant tests and documentation so they no longer promise public
  npm installation or update discovery.
- R8: Treat the fork checkout as the sole distribution source for its single
  maintainer; no automatic source pull, package installation, or release-asset
  download is required.
- R9: Do not run a Trellis update against this repository, modify its root
  `.trellis/.version`, or migrate its existing `0.6.2` dogfood deployment.
  Validation must use isolated fixtures for projects receiving the fork.
- R10: Support a one-time upgrade for existing consumer projects that carry an
  upstream-versioned `0.6.x` stamp into the first fork-owned release.

## Acceptance Criteria

- [ ] No command or generated session-context path queries the public npm
  registry or names `@mindfoldhq/trellis` as its upgrade source.
- [ ] A project generated by the fork can reliably detect whether its
  `.trellis/.version` differs from the fork CLI that is actually running.
- [ ] `trellis update` retains its existing refusal to downgrade by default and
  applies the ordered historical `0.6.x` ingress chain once before continuing
  on fork-owned versions.
- [ ] Core and CLI still fail release validation when their fork versions differ.
- [ ] The repository records the upstream baseline and selected imports without
  treating upstream release numbers as the fork compatibility version.
- [ ] Release automation no longer attempts public npm publishing or npm
  visibility checks.
- [ ] Focused regression tests cover local mismatch detection, no-network
  update behavior, and fork release/migration invariants.
- [ ] The repository's root `.trellis/.version` remains `0.6.2`; no task step
  invokes an update or migration against this repository as a consumer.
- [ ] An isolated consumer fixture starting from an upstream `0.6.x` project
  stamp upgrades once to the first fork version without a downgrade error and
  applies the expected migration chain exactly once.
- [ ] The former `upgrade` path neither invokes npm nor mutates the source
  checkout; its replacement behavior clearly directs the maintainer to update
  the checkout and rebuild the CLI outside the command.

## Out of Scope

- Mirroring every upstream feature or release.
- Publishing either package to the public npm registry.
- Changing unrelated Trellis workflow behavior.

## Key Decisions

- The fork is currently used only by its maintainer from a source checkout.
- There is no package publication, private registry, release-asset channel, or
  automatic self-update requirement.
- Upstream changes may be selectively synchronized; their provenance remains
  separate from the fork compatibility version.
- Existing consumer projects with upstream `0.6.x` stamps must be able to
  perform a one-time upgrade into the fork-owned version line.
- `1.0.0` is the first fork compatibility release. It is a target version for
  consumer projects only; it does not migrate this repository's dogfood
  `.trellis/.version` stamp.

## Deferred Items

- A future multi-user or packaged distribution may introduce a deliberate
  delivery channel. It is not part of this task and must not silently restore
  public npm behavior.
